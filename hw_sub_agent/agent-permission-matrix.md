# Agent 权限矩阵对比分析

## 权限矩阵总览

| 权限     | 采集 Agent (collector) | 分析 Agent (analyzer) | 整理 Agent (organizer) |
| -------- | :--------------------: | :-------------------: | :--------------------: |
| Read     |           ✓            |           ✓           |           ✓            |
| Grep     |           ✓            |           ✓           |           ✓            |
| Glob     |           ✓            |           ✓           |           ✓            |
| WebFetch |           ✓            |           ✓           |           ✗            |
| Write    |           ✗            |           ✗           |           ✓            |
| Edit     |           ✗            |           ✗           |           ✓            |
| Bash     |           ✗            |           ✗           |           ✗            |

---

## 逐 Agent 分析

### 采集 Agent (collector)

**角色**：数据入口 —— 从外部源抓取原始信息。

| 权限 | 原因 |
|------|------|
| WebFetch ✓ | 核心能力。需要访问 GitHub Trending 和 Hacker News 获取内容。 |
| Write ✗ | 不直接写文件。采集结果通过对话返回给编排引擎，由引擎决定是否及何时落盘到 `knowledge/raw/`。避免脏数据直接进入文件系统。 |
| Bash ✗ | 无执行外部命令的需求，任务是获取和结构化网页内容。 |

**定位**：纯输入端，靠近外部世界，远离文件系统。

---

### 分析 Agent (analyzer)

**角色**：智能加工 —— 读原始数据，生成摘要、标签、评分。

| 权限 | 原因 |
|------|------|
| WebFetch ✓ | 需要访问条目的原始 URL（GitHub README、HN 帖子正文）获取完整信息，确保摘要不编造。 |
| Write ✗ | 分析结果同样是半成品——可能在整理阶段被去重、合并、降级。直接写盘会导致不合格数据残留。 |
| Bash ✗ | 无执行外部命令的需求，任务是基于文本内容进行推理和结构化。 |

**定位**：纯分析层，有外部信息获取能力，但输出不走文件系统。

---

### 整理 Agent (organizer)

**角色**：数据质量守门员 —— 校验、去重、格式化、落盘。

| 权限 | 原因 |
|------|------|
| **Write ✓** | 整个流水线中唯一的写入节点。经过分析 Agent 产出的数据，由整理 Agent 完成最终校验（格式、去重、命名）后才写入 `knowledge/articles/`。集中写入权限确保数据质量的最后一道关由单一角色负责，避免多 Agent 写入导致的数据不一致。 |
| **Edit ✓** | 去重合并场景下需要修改已有文件（如将重复条目的标签追加到主条目）。 |
| **WebFetch ✗** | 此为核心设计决策，详见下文。 |
| Bash ✗ | 无执行外部命令的需求，任务是基于已有数据的校验和 I/O 操作。 |

**定位**：纯输出端，有文件系统写入能力，但与外部网络隔离。

---

## 核心设计：为什么整理 Agent 有 Write 但没有 WebFetch？

### 一句话回答

**外部数据获取与本地持久化分属两个安全域，不应由同一 Agent 承担。**

### 分层隔离原则

```
┌──────────────────────────────────────────────────────┐
│                     外部网络（不可信）                   │
│  ┌──────────────┐  ┌───────────────┐                   │
│  │  GitHub       │  │  Hacker News   │                   │
│  └──────┬───────┘  └───────┬───────┘                   │
└─────────┼──────────────────┼───────────────────────────┘
          │ WebFetch         │ WebFetch
          ▼                  ▼
┌──────────────────────────────────────────────────────┐
│                  Agent 层（只读域）                      │
│  ┌──────────┐     ┌───────────┐                        │
│  │ Collector │────▶│ Analyzer   │                        │
│  │           │     │           │                        │
│  │ WebFetch✓ │     │ WebFetch✓ │                        │
│  │ Write   ✗ │     │ Write   ✗ │                        │
│  └──────────┘     └─────┬─────┘                        │
└─────────────────────────┼──────────────────────────────┘
                          │ 结构化 JSON（通过编排引擎传递）
                          ▼
┌──────────────────────────────────────────────────────┐
│                  Agent 层（读写域）                      │
│  ┌──────────────────────┐                              │
│  │     Organizer         │                              │
│  │                      │                              │
│  │  Write   ✓ (唯一写者)  │                              │
│  │  WebFetch ✗          │                              │
│  └──────────┬───────────┘                              │
└─────────────┼──────────────────────────────────────────┘
              │ Write
              ▼
┌──────────────────────────────────────────────────────┐
│                 文件系统（受信域）                        │
│  knowledge/articles/*.json                             │
└──────────────────────────────────────────────────────┘
```

### 三个维度展开

#### 1. 安全隔离：防止"提权写入"

```
危险场景（禁止）：
  Organizer 有 WebFetch + Write
  → 访问恶意站点 → 获取注入内容 → 直接写入知识库
  → 攻击面扩大，无校验关口

安全设计（当前）：
  Collector/Analyzer 有 WebFetch 但无 Write
  → 获取到的数据只能输出，无法落盘
  → 必须经过编排引擎传递
  → Organizer 仅处理已被分析 Agent「消化」过的结构化数据
  → Organizer 再写入 → 数据全程都在可审计链路内
```

Organizer 拿到的是已经过分析 Agent 处理的 JSON——字段齐全、评分已判、摘要已生成。它不再需要访问原始 URL，因为所有需要从外部获取的信息已经在分析阶段完成了。

#### 2. 职责纯粹性：Organizer 的"输入合约"

```
Organizer 的输入合约：
  - 所有字段已由 Analyzer 填充
  - source_url 已存在，无需再次抓取
  - summary / tags / score 已完成，无需修改内容

Organizer 的工作：
  - 校验字段格式（非空、类型、范围）
  - 去重（URL 比对）
  - 命名 + 落盘
  ── 这些操作 100% 是本地操作，不需要网络 ──
```

给 Organizer WebFetch 是过度授权——它根本不需要，也不应该用它。

#### 3. 审计链：数据溯源清晰

```
每条 knowledge/articles/*.json 的溯源路径：

  GitHub/HN 页面 ──WebFetch──▶ Collector ──输出──▶ Engine
                                                  │
  source_url   ──WebFetch──▶ Analyzer  ──输出──▶ Engine
                                                  │
                                      ──传递──▶ Organizer ──Write──▶ 文件

  问题排查时：
  - 摘要不准确？    → 回溯 Analyzer 的分析过程
  - 原始信息错误？  → 回溯 Collector 的抓取过程
  - 文件格式不对？  → 回溯 Organizer 的校验日志
  - 恶意内容入库？  → 唯一入口是 Organizer.Write，追溯简单
```

## 三类 Agent 权限设计的统一逻辑

| 维度           | Collector       | Analyzer        | Organizer       |
| -------------- | --------------- | --------------- | --------------- |
| 网络（WebFetch） | 数据入口，必须   | 深入阅读，必须    | 数据已完备，禁止   |
| 文件（Write）    | 输出给引擎，不写  | 输出给引擎，不写   | 最终守门，唯一写   |
| Shell（Bash）   | 无需，禁止       | 无需，禁止        | 无需，禁止        |

核心原则：**横向切断 WebFetch + Write 组合权限，纵向按流水线阶段分配最小权限。**
